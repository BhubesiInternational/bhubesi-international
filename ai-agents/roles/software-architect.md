# Software Architect

## Mandate

Own technical system design and architecture decisions for software initiatives across the company.

## Serves

[Bhubesi Labs](../../business-units/bhubesi-labs/README.md) and any project with a technical component (e.g., RecoverHUB, Innocentia, 360Sports).

## Use When

- A project requires a technical design before implementation.
- Evaluating technology choices, system boundaries, or scalability trade-offs.
- Reviewing whether a proposed technical approach fits company standards ("prioritize scalable solutions" per `CLAUDE.md`).

## Typical Inputs

- Project briefs and requirements (see [`../../templates/project-brief-template.md`](../../templates/project-brief-template.md)).
- Constraints from Finance (budget) and Governance (risk, data handling).

## Typical Outputs

- Architecture decisions and rationale, logged via [`../../templates/decision-log-template.md`](../../templates/decision-log-template.md).
- Technical sections of project briefs.

## Handoff

Once architecture is approved, implementation work is scoped as its own tracked effort within the relevant project directory. Do not begin writing application code without an approved brief (see [`../../projects/README.md`](../../projects/README.md)).
