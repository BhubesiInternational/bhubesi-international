# Project Kickoff Workflow

The process for taking an idea from proposal to an active, approved project (see [`../projects/README.md`](../projects/README.md) for the project lifecycle overview).

## Steps

1. **Draft the brief.** Use [`../templates/project-brief-template.md`](../templates/project-brief-template.md) to capture the problem, target outcome, scope, lead business unit, and resourcing needs.
2. **Identify supporting roles.** Use [`../ai-agents/README.md`](../ai-agents/README.md) to determine which executive roles should contribute (e.g., Business Strategist for market fit, Software Architect for technical projects, Financial Analyst for budget).
3. **Route for approval.** Per [`../docs/governance.md`](../docs/governance.md), Strategy reviews all new projects; Finance reviews budget; Legal reviews any external commitments or IP considerations.
4. **Assign a lead.** Every approved project needs a named lead (human) responsible for delivery, supported by the Project Manager role where useful.
5. **Create the project directory.** Add `projects/<name>/README.md` following the pattern of existing project READMEs (mandate, status, lead business unit, next steps).
6. **Set milestones.** Define the first checkpoint(s) at which the project will be reviewed.
7. **Execute per the standard workflow.** See [`standard-workflow.md`](./standard-workflow.md).

## Graduation or Closure

At each milestone, decide: continue, graduate into a business unit's ongoing responsibility, or close. Document the decision via [`../templates/decision-log-template.md`](../templates/decision-log-template.md) regardless of outcome — closed or paused projects are still part of the company's record.
